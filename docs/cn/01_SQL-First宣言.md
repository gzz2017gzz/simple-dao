## 序：框架的终点是消失

**技术的最佳状态，是让人感觉不到它的存在。**

好的门把手不会告诉你的手该怎么握，它只是在那里，你一推，门就开了。

好的数据库访问层也应该是这样：你心里想的是“我要查用户和他的部门”，手指敲出来的是 SQL，回车，数据回来。

没有 XML 要配，没有 Mapper 要写，没有 `@OneToMany` 要琢磨，没有 N+1 问题要救火。

**这就是 SQL-First 范式。**

它不是一个新的框架。它是一个持续了50年、却被遗忘了20年的答案。

---

### 评价技术的终极尺度
我们如何评判一个技术的好坏？不是看它有多“高级”、功能有多全，而是看它是否**减小了人的心智负担**，是否**降低了使用成本**。最基本的底线：它得帮你少写代码，而不是让你为它写更多代码。

**业务代码浓度越高越好，框架存在感越低越好。** 框架应当像空气，存在但不觉；应当像助手，帮忙但不添乱。

**不作恶是最大的善，不添乱是最大的善。** 框架最大的恶，就是制造原本不存在的麻烦。

## 第一章：三十年弯路——ORM 如何从“解决方案”变成“问题本身”

### 1.1 那个被遗忘的共识
1974年，IBM 的 Donald Chamberlin 和 Raymond Boyce 发明了 SQL。它的设计目标极其明确：**让人类用最接近自然语言的方式，操作关系模型。**

从那一天起，数据库就有了它的母语。不是 Java，不是 Python，不是 C#。是 SQL。

### 1.2 阻抗不匹配：一个被制造出来的问题
1990年代，对象导向编程席卷业界。开发者看着数据库里的表和行，再看代码里的类和对象，产生了一种焦虑：**“它们长得不一样。”**

这种焦虑被命名为“阻抗不匹配”。然后一个庞大的产业诞生了：ORM 框架、培训课程、技术书籍、咨询业务……**但没人停下来问：这个沟，真的存在吗？**

### 1.3 ORM 的三个原罪
**原罪一：制造方言**  
SQL 是国际标准。ORM 非要发明自己的方言：JPQL/HQL、Criteria API、MyBatis 动态 SQL。每个方言都需要专门学习，学习成果无法迁移到下个项目、下一种语言。

**原罪二：扭曲语义**  
关系数据库的核心是**集合**。ORM 的核心是**对象图**。这种扭曲导致 N+1 查询、笛卡尔积爆炸、缓存策略复杂化。

**原罪三：转移错误**  
原本的错误只有两种：SQL 写错了、参数传错了。ORM 引入第三类错误：**框架自造错误**。MyBatis 有 31 类，JPA 有无头悬案。开发者把 30% 的时间花在解决“框架问题”上，而不是业务问题。

## 第二章：SQL-First 范式——回归本质的四条法则

### 2.1 法则一：单表操作对象化，但止步于对象化
**继承即得，零代码。**
```java
@Repository
public class UserDao extends BaseDao<User> {
    // 空类，获得所有单表 CRUD
}
```
**边界清晰**：自动生成 INSERT/UPDATE/DELETE/SELECT BY ID，自动填充审计字段，自动映射驼峰/下划线。**到此为止**。不做级联、懒加载、自动 flush。

### 2.2 法则二：联表查询 SQL 化，不扭曲、不包装、不发明方言
**SQL 是联表查询的唯一正确表达。**
```java
private static final String SQL = """
    SELECT u.*, d.name AS dept_name
    FROM user u LEFT JOIN dept d ON u.dept_id = d.id
    """;
public Page<UserVO> pageJoin(UserCond cond) {
    return page(SQL, cond, UserVO.class);
}
```
特性：IDE 语法高亮、可直接复制到数据库客户端执行、执行计划分析零干扰。

### 2.3 法则三：条件拼接极简化，用母语代替方言
**BaseCondition：不是查询构建器，是 SQL 片段拼接器。**
```java
public class UserCond extends BaseCondition {
    @Override
    protected void addCondition() {
        and("name LIKE ?", name, 3);
        and("age > ?", ageMin);
        add("AND d.dept_name LIKE ?", deptName, 3);
        add("AND EXISTS (SELECT 1 FROM order o WHERE o.user_id = t.id)", hasOrder);
    }
}
```
**因为 SQL 你本来就会。**

### 2.4 法则四：单表/联表共用同一套条件工具，无割裂、无切换
同一个 Cond 类，同一个 addCondition() 方法，同一套参数收集机制。开发者不需要学习两套 API，不需要切换思维模式，不需要维护两份条件逻辑。**统一，是比“简洁”更高级的简洁。**

## 第三章：SQL优化主权——谁控制SQL，谁就控制性能

**SQL优化主权**：开发团队对最终执行 SQL 的**可见、可理解、可修改、可优化**的能力。

| 框架 | 主权等级 | 表现 |
|------|--------|------|
| JPA/Hibernate | ❌ 完全丧失 | SQL 由框架生成，只能通过 hints 间接影响 |
| MyBatis | ⚠️ 半遮半掩 | SQL 可见，但被 XML 碎片化，优化需先拼完整 SQL |
| SQL-First | ✅ 完全掌控 | SQL 完整直写，可直接 explain、加索引、改写 |

一个真实案例：12 表联查 + 60 个动态条件，响应时间要求 1 秒内。  
- JPA：不可行  
- MyBatis：勉强可行，但优化链路漫长  
- SQL-First：复制完整 SQL → explain → 改索引 → 贴回代码，**优化链路缩短 80%**

## 第四章：不添乱即大善——框架的终极道德

### 4.1 框架的道德哲学
- **不伤害原则**：框架不应制造原本不存在的错误。
- **透明原则**：框架不应隐藏核心执行路径。
- **责任清晰原则**：SQL 的问题归 SQL，参数的问题归参数，框架的问题归框架——但框架不应该有自己的问题。

### 4.2 善的三重境界
**第一重：解决问题**（JdbcTemplate）  
**第二重：不制造问题**（Spring JDBC）  
**第三重：让解决问题的人感到被尊重**（SimpleDAO：把 SQL 优化主权交还给开发者）

### 4.3 扩展必须白盒
当需要扩展功能（如数据权限、分页、脱敏）时，框架应当提供**白盒**方式，让开发者用业务代码直接实现，而不是强迫开发者深入框架底层扒拦截器、改写执行链路。MyBatis 的生态之所以臃肿，正是因为其核心执行黑盒，导致分页、脱敏、权限等功能都必须通过拦截器暴力修改 SQL，20 行代码摸不到业务的边儿，版本升级就崩。

**Java 持久层框架整体糟糕，唯独 Spring JDBC 做得稍好**——它让 SQL 执行白盒化，业务扩展无门槛。SimpleDAO 站在 Spring JDBC 的肩膀上，将这种白盒哲学贯彻到底。

## 终章：可移植的思想——200 行代码改变世界

SQL-First 不是 Java 的特权。任何语言都有自己的“Spring JDBC 等价物”：Python 的 `sqlite3`、Node 的 `mysql2`、Go 的 `database/sql`……200-300 行核心代码，即可搭建该语言的 SQL-First 数据访问层。

思想的传播不需要组织。下一个 Python 开发者受够了 SQLAlchemy 的复杂，会找到这里；下一个 Node 开发者在 TypeORM 的 issue 里挣扎，会找到这里；下一个 Go 程序员困惑“为什么 GORM 把简单查询搞得这么重”，会找到这里。他们会自己写 pydao、node-sql-first、gosqlc。**他们不需要认识你。他们只需要读到这篇宣言。**

## 附录：常见误读澄清
- **Q：SQL-First 是反对 ORM 吗？**  
  A：是，也不是。反对 ORM 试图“替代 SQL”的那部分野心；单表对象化恰恰是 ORM 做对的事情。
- **Q：SQL-First 适合所有项目吗？**  
  A：不适合需要把数据库完全抽象掉的项目。但如果你需要掌控性能、需要维护复杂业务逻辑，SQL-First 是最诚实的选择。
- **Q：手写 SQL 会不会导致 SQL 注入？**  
  A：用预编译占位符（? 或 :name），不会。所有 ORM 最终也是用预编译。
- **Q：这是不是重复造轮子？**  
  A：如果你认为“把 SQL 放回原位”是造轮子，那么是的。但有些轮子，值得反复确认它确实是圆的。
