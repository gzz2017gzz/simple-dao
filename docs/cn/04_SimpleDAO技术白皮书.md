# 04 SimpleDAO技术白皮书
## 全场景碾压ORM框架白皮书

核心定位：不是 ORM 框架的替代方案，是数据库访问层的**认知重构与范式革命**，以 Spring JDBC 原生能力为内核，以「框架克制、回归本质、不添乱即大善」为设计哲学，实现单表自动化+联表原生化+复杂SQL自由化，将 SQL 优化的完整主动权交还给开发者，最终实现**开发者的时间解放**。

---

## 一、核心结论：回归本质，让框架做「隐形助手」才是终极解

SimpleDAO 的核心价值，是对数据库访问层框架设计的**底层逻辑纠偏**与**行业认知重构**：摒弃 ORM 框架「为抽象而抽象、为封装而封装」的误区，拒绝制造框架私有规则、不产生自造错误，仅做**样板代码的消除者、数据库错误的传递者、业务逻辑的守护者、开发者杂事的分担者**。

**评价技术的终极尺度**：是否减小了人的心智负担？是否降低了使用成本？最基本的底线：它得帮你少写代码，而不是让你为它写更多代码。  
**业务代码浓度越高越好，框架存在感越低越好。** 框架最大的善，是不作恶、不添乱。

在所有技术场景下，SimpleDAO 均实现对 MyBatis/JPA 的碾压性优势；仅在「遗留系统渐进迁移」「团队深度依赖旧框架」等非技术场景，MyBatis 具备过渡性价值，而这一价值，本质是行业集体认知惯性的产物，而非技术本身的优越性。

**框架设计的终极真理**：好的框架应像空气一样存在，默默提供支持却从未成为开发阻碍；**不造错、不添乱，是框架对开发者最顶级的善**。

## 四、MyBatis现象底层解构

### 1. 核心原罪：错误转移术
MyBatis 将原生的业务/SQL错误，转移为框架专属的配置/映射/语法错误，让开发者的工作重心从“解决业务问题”变成“伺候框架问题”。

### 2. 关键伪装：技术债务的显性化包装
MyBatis 将所有自造错误包装成“核心功能特性”，让开发者陷入“复杂=高级、冗余=灵活”的认知误区。

### 3. 生态闭环：受害者联盟的自我强化
框架制造问题 → 开发者踩坑 → 社区提供解决方案 → 方案产生新问题 → 继续踩坑。

### 4. 行业根源：集体认知失调
开发者同时面临两个矛盾认知：实际体验差 vs 社会认知认为它是行业标准。为减少心理冲突，选择自我说服，陷入惯性依赖。

### 5. 黑盒执行的代价
MyBatis 的 SQL 执行链路是黑盒，导致分页、脱敏、数据权限等扩展无法用业务代码实现，只能通过拦截器暴力修改 SQL。写一个数据权限，20 行代码全是框架底层 API，版本升级必崩。  
相比之下，**Spring JDBC 是白盒**，SQL 执行透明，开发者可以在 Service 层用 AOP 轻松实现扩展，代码全是业务逻辑。SimpleDAO 继承这一优点，让扩展回归业务。

## 五、SQL优化主权争夺战

- **JPA/Hibernate**：纯纯绊脚石，负向作用拉满  
- **MyBatis**：半遮半掩的枷锁，不阻碍但添乱  
- **SimpleDAO**：零阻碍、纯赋能，让开发者聚焦 SQL 本身

**核心反差**：ORM 框架把开发者的时间消耗在框架问题与无意义的工序中，SimpleDAO 把这些时间全部还给开发者。

## 六、SimpleDAO 核心优势拆解

### 1. 单表操作：极致简洁，零冗余代码
- 实现：基于 Spring JdbcTemplate 封装 BaseDao，自动实现所有单表 CRUD  
- 案例：`ChannelDao extends BaseDao<Channel>` 一行代码实现所有操作  
- 错误消除：干掉 Mapper 接口、XML 文件、手动 SQL

### 2. 联表/复杂SQL：SQL自由，无框架束缚
- 设计：直接使用原生 SQL 编写，搭配 BaseCondition 做动态条件拼接  
- 案例：12 表联查（GradeDao）、报表级 SQL（IncomeDao）  
- 错误消除：摒弃 XML 片段拆分、resultMap 嵌套

### 3. 动态SQL：Java原生逻辑，零学习成本
- 实现：BaseCondition 多重载 add() 方法，覆盖所有动态 SQL 场景  
- 案例：GradeCond 支持 80+ 动态查询条件  
- 错误消除：干掉 OGNL 表达式、XML 动态标签

### 4. 数据权限：注解驱动，优雅无侵入
- 实现：@BusinessAuth 注解 + AOP 拦截  
- 案例：一行注解实现多维度数据权限控制  
- 错误消除：摒弃自定义 StatementHandler 拦截器

### 5. 对象映射：自动适配，零配置
- 实现：内置驼峰/下划线转换，搭配 @Table/@Id/@Column  
- 错误消除：干掉手动 resultMap 配置

### 6. 调试排错：透明化传递，分钟级定位
- 设计：不封装、不转换、不隐藏数据库原生错误  
- 技巧：Sql.fill() 打印带完整参数的执行 SQL  
- 错误消除：摒弃 MyBatis 的错误包装与转换

### 7. 分布式/批量：原生能力，性能可控
- 分布式分库：手写分片 SQL，结合 BaseCondition 拼接  
- 批量操作：复用 Spring JdbcTemplate 的 batchUpdate  
- 错误消除：摒弃分库配置错误、批量插件性能问题

## 七、常见质疑全维度反驳

**质疑1：SimpleDAO 生态弱，遇到问题没解决方案？**  
反驳：MyBatis 的生态是解决框架自造问题的“受害者联盟”生态。SimpleDAO 基于 Spring 原生生态构建，开发者遇到的问题均为通用问题，无需依赖社区。

**质疑2：缺乏 ORM 高级功能（懒加载/级联操作）？**  
反驳：懒加载、级联操作是伪需求。SimpleDAO 用原生 SQL 关联查询替代，更清晰可控。

**质疑3：SQL 写在 Java 里，耦合度高？**  
反驳：MyBatis 的“SQL与Java分离”是割裂而非解耦。SimpleDAO 的一站式开发效率更高，且 SQL 保持完整。

**质疑4：手写 SQL 增加开发工作量？**  
反驳：ORM 代码生成器仅在初期简单 CRUD 阶段有微弱优势。企业级开发的核心痛点是复杂联表、动态条件、性能优化，手写 SQL 的效率远高于维护 XML+映射+配置的长期成本。

**质疑5：学习成本低，是不是功能简单？**  
反驳：学习成本低是因为不发明新概念。SimpleDAO 已在生产环境稳定运行 3 年+，支撑日均百万级请求，覆盖复杂场景。

## 八、适用场景与落地建议

**优先选择 SimpleDAO 场景**：新项目开发、复杂业务系统、高并发/高性能场景、分布式分库分表、团队技术标准化。

**MyBatis 过渡保留场景**：遗留系统大规模迁移、团队深度依赖 MyBatis、第三方组件强依赖。

**落地实施建议**：  
1. 引入核心封装（1小时）  
2. 新功能全量落地（1周适应）  
3. 旧功能渐进替换（零风险）  
4. 团队零培训落地

## 九、框架设计的终极哲学：不添乱即大善

**SimpleDAO 的四大核心设计原则**：  
1. 不发明新概念  
2. 不制造新错误  
3. 不隐藏核心逻辑  
4. 只解决真问题，只分担杂事

**技术演进的真相**：前进，是勇敢的减法，是对开发者时间的珍惜。  
**终极价值**：技术的善，是让敲代码的人，有更多时间享受代码之外的生活。
