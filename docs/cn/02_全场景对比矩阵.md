# 02 全场景对比矩阵
## ORM vs SQL-First，一图终结争论

版本：1.0  
发布日期：2026年2月12日  
对比对象：JPA/Hibernate、MyBatis、SQL-First（SimpleDAO范式）  
对比原则：可验证、可重现、不双标

## 一、总表：21个维度全景对比

| 维度 | JPA/Hibernate | MyBatis | **SQL-First范式** | 结论 |
|------|---------------|---------|-------------------|------|
| **1. 单表CRUD代码量** | 中等 | 高 | **极低（继承BaseDao，空类）** | SQL-First ↓80% |

## 二、关键维度深度拆解

### 2.1 单表CRUD：从“写代码”到“声明需求”
**JPA**：继承黑盒，不知道 SQL 怎么生成。  
**MyBatis**：一个查询，两个文件，三个步骤。  
**SQL-First**：继承 BaseDao，空类获得所有单表方法。

### 2.2 复杂联表：框架的试金石
**测试用例：8表联查 + 20个动态条件**  
**MyBatis**：约180行 XML，SQL 被拆成4个片段，调试需拼完整 SQL。  
**SQL-First**：完整70行 SQL，复制到 Navicat 直接执行，优化链路 1 分钟。

### 2.3 动态条件：母语 vs 方言
**需求**：if name != null, add name like query; if ageMin != null, add age > ageMin  
**JPA Criteria**：语义倒置，不可读。  
**MyBatis**：OGNL 语法、标签闭合、字符串拼接。  
**SQL-First**：`and("name LIKE ?", name, 3); and("age > ?", ageMin);`

### 2.4 扩展白盒性：MyBatis 生态为何臃肿
MyBatis 的核心执行是黑盒，导致分页、脱敏、数据权限等扩展功能无法用业务代码直接实现，只能通过拦截器暴力修改 SQL 执行链路。写一个数据权限，20 行代码摸不到业务的边儿，全是框架底层 API，版本升级就崩。  
**Spring JDBC 则是白盒**，SQL 执行透明，开发者可以在 Service 层用 AOP 轻松实现扩展。SimpleDAO 继承这一优点，让扩展回归业务。

### 2.5 SQL优化主权：谁控制了SQL，谁就控制了性能
- **JPA**：夺走了你的 SQL  
- **MyBatis**：借走了你的 SQL  
- **SQL-First**：把 SQL 还给了你

## 三、社区本质：受害者联盟 vs 无社区

MyBatis 社区的 issue 分布：  
- XML 配置错误 25%  
- ResultMap 映射失败 20%  
- OGNL 表达式错误 15%  
- 插件冲突 10%  
- 版本兼容问题 10%  
- 真正的 SQL/业务问题 **20%**

**80% 的社区讨论，是在解决框架制造的问题。**  
这就是“受害者联盟”：框架制造问题 → 开发者踩坑 → 社区提供解决方案 → 方案产生新问题 → 继续踩坑。

SQL-First 的 issue：  
- 这个 SQL 查询慢 → 数据库/SQL 问题  
- 事务不回滚 → Spring 事务问题  
- 参数传错了 → 业务代码问题  

**没有一个问题是“框架问题”。** 工具不需要社区。你用锤子砸到手，不会去锤子社区发帖。

## 四、误解澄清
- **误解1：SQL-First = 回到 JDBC 原始社会**  
  澄清：你见过会自己生成 INSERT、自动填充主键、自动分页的“原始社会”？
- **误解2：手写 SQL 容易注入**  
  澄清：注入来自字符串拼接，不是来自手写。SQL-First 强制预编译。
- **误解3：复杂映射必须用 resultMap**  
  澄清：80% 的复杂映射源于试图用 ORM 做数据库不该做的事。剩余 20% 手写代码更清晰。

                           │
                           │
        MyBatis（半遮半掩）●──┬──● JPA（完全丧失）
                           │
                           │
                           │
                     框架自造错误
