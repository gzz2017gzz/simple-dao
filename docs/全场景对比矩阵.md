
# 全场景对比矩阵
**——ORM vs SQL-First，一图终结争论**

版本：1.0  
发布日期：2026年2月12日  
对比对象：JPA/Hibernate、MyBatis、SQL-First（SimpleDAO范式）  
对比原则：可验证、可重现、不双标

## 一、总表：20个维度全景对比

| 维度 | JPA/Hibernate | MyBatis | **SQL-First范式** | 结论 |
|------|---------------|---------|-------------------|------|
| **1. 单表CRUD代码量** | 中等（Repository+实体注解） | 高（Mapper+XML+SQL） | **极低（继承BaseDao，空类）** | SQL-First ↓80% |
| **2. 单表CRUD配置量** | 中（@Entity/@Id/@GeneratedValue） | 高（XML namespace/resultMap） | **零（@Table/@Id两注解）** | SQL-First ↓100% |
| **3. 轻量联表（2-3表）** | 可（JPQL）但N+1风险 | 繁琐（XML+resultMap） | **极简（原生SQL+自动映射）** | SQL-First ★★★★★ |
| **4. 复杂联表（5+表）** | ❌ 不可维护（JPQL爆炸） | ⚠️ 勉强（XML碎片地狱） | **✅ 清晰（完整SQL直写）** | SQL-First 唯一解 |
| **5. 报表级SQL（UNION/子查询）** | ❌ 完全不支持 | ⚠️ 极其痛苦（标签嵌套） | **✅ 原生支持+参数合并** | SQL-First 碾压 |
| **6. 动态SQL** | Criteria API（语义倒置） | OGNL+XML标签（方言） | **Java/Python/Go原生if** | SQL-First 零学习成本 |
| **7. 条件构建统一性** | ❌ 单表/复杂查询两套 | ❌ MP单表Wrapper，联表XML | **✅ 完全统一（BaseCondition）** | SQL-First 独家 |
| **8. 对象映射** | 自动但黑盒 | 手动resultMap | **自动（驼峰/下划线）** | SQL-First 够用且透明 |
| **9. 数据权限** | 需拦截器/过滤器 | 自定义StatementHandler | **注解+AOP，一行代码** | SQL-First 优雅 |
| **10. 批量操作** | batch API复杂 | 需第三方插件 | **原生batchUpdate复用** | SQL-First 零依赖 |
| **11. 分页** | PageRequest（方言） | PageHelper（插件） | **内置page(cond)方法** | SQL-First 内置 |
| **12. 事务** | @Transactional | 需额外配置事务管理器 | **@Transactional原生支持** | SQL-First 零适配 |
| **13. 多数据源** | 需额外配置 | 需额外配置 | **Spring原生支持** | SQL-First 零成本 |
| **14. 分布式分库** | ShardingSphere适配 | ShardingSphere+XML | **手写分片SQL，清晰可控** | SQL-First 主权在我 |
| **15. SQL调试** | 黑盒，需开启日志 | 需解析BoundSql | **断点直接看完整SQL** | SQL-First 分钟级排错 |
| **16. 错误来源** | 框架错误+业务错误 | **22类框架自造错误**+业务错误 | **仅数据库/业务错误** | SQL-First 不添乱 |
| **17. SQL优化主权** | ❌ 完全丧失 | ⚠️ 半遮半掩 | **✅ 完全掌控** | SQL-First 本质优势 |
| **18. 学习成本** | 高（缓存/级联/生命周期） | 中高（XML/OGNL/映射） | **极低（会SQL就会）** | SQL-First ↓90% |
| **19. 知识可迁移性** | ❌ 仅JPA生态 | ❌ 仅MyBatis生态 | **✅ 通用SQL技能** | SQL-First 终身受益 |
| **20. 社区本质** | 规范讨论组 | **受害者联盟** | **无社区（没问题可问）** | SQL-First 健康态 |

## 二、关键维度深度拆解

### 2.1 单表CRUD：从“写代码”到“声明需求”

**JPA：**
```java
@Repository
public interface UserRepository extends JpaRepository<User, Long> {
    // 自动方法：findById、save、delete...
}
```
*问题：继承的是黑盒，不知道SQL怎么生成，N+1随时暴雷。*

**MyBatis：**
```java
@Mapper
public interface UserMapper {
    User selectById(Long id);  // 还需写XML
}
```
```xml
<select id="selectById" resultType="User">
    SELECT * FROM user WHERE id = #{id}
</select>
```
*问题：一个查询，两个文件，三个步骤。*

**SQL-First：**
```java
@Repository
public class UserDao extends BaseDao<User> {
    // 空类，所有单表方法已就绪
}

userDao.findById(1L);        // 直接用
userDao.save(user);          // 直接用
userDao.page(cond);          // 直接用
```
*本质：从“我在调用框架的方法”到“工具在替我完成”。*

---

### 2.2 复杂联表：框架的试金石

**测试用例：8表联查 + 20个动态条件**

**JPA：** 不可行。放弃。

**MyBatis：**

```xml
<!-- 约180行，含： -->
<resultMap id="xxx" extends="yyy">
    <association .../>
    <collection .../>
</resultMap>
<sql id="base_column">...</sql>
<sql id="join_clause">...</sql>
<select id="pageJoin" resultMap="xxx">
    <include refid="base_column"/>
    <include refid="join_clause"/>
    <where>
        <if test="name != null">AND t.name = #{name}</if>
        <if test="deptId != null">AND d.id = #{deptId}</if>
        <!-- 20个if -->
    </where>
</select>
```

**痛点：**
- SQL被拆成4个片段，可读性崩溃
- 调试需开日志拼完整SQL
- 加一个字段要改3个地方

**SQL-First：**

```java
private static final String SQL = """
    SELECT t.*, d.name dept_name, ...  -- 完整70行SQL
    FROM main_table t
    LEFT JOIN dept d ON ...
    -- 8个JOIN完整可见
    """;

public Page<Vo> pageJoin(Cond cond) {
    return page(SQL, cond, Vo.class);
}
```

**优势：**
- SQL完整，IDE语法高亮
- 复制到Navicat直接执行
- 优化链路：复制 → explain → 改索引 → 贴回 → **1分钟**

### 2.3 动态条件：母语 vs 方言

**需求：** if name != null, add name like query; if ageMin != null, add age > ageMin

**JPA Criteria：**
```java
Predicate p = cb.conjunction();
if (name != null) {
    p = cb.and(p, cb.like(root.get("name"), "%" + name + "%"));
}
if (ageMin != null) {
    p = cb.and(p, cb.greaterThan(root.get("age"), ageMin));
}
query.where(p);
```
*问题：语义倒置（谓语在前，条件在后），不可读。*

**MyBatis：**
```xml
<if test="name != null and name != ''">
    AND name LIKE CONCAT('%', #{name}, '%')
</if>
<if test="ageMin != null">
    AND age > #{ageMin}
</if>
```
*问题：OGNL语法、标签闭合、字符串拼接。*

**SQL-First：**
```java
and("name LIKE ?", name, 3);  // 3=全模糊
and("age > ?", ageMin);
```
*本质：这就是SQL，只是把值换成占位符。*

### 2.4 SQL优化主权：谁控制了SQL，谁就控制了性能

| 场景 | JPA | MyBatis | SQL-First |
|------|-----|---------|-----------|
| 看到最终执行SQL | ❌ 需开日志，难解析 | ⚠️ 需开日志，难阅读 | ✅ 断点直接看 |
| 修改SQL | ❌ 改JPQL，不确定生成效果 | ⚠️ 改XML片段，需拼完整 | ✅ 直接改完整SQL |
| 加索引建议 | ❌ 猜 | ⚠️ 猜+验证 | ✅ 执行计划，精准定位 |
| 复杂SQL优化 | ❌ 无解 | ⚠️ 拆碎-拼接-优化-拆碎 | ✅ 完整SQL，一次优化 |

**一句话结论：**

**JPA夺走了你的SQL，MyBatis借走了你的SQL，SQL-First把SQL还给了你。**

## 三、社区本质：受害者联盟 vs 无社区

### 3.1 MyBatis社区的“繁荣”真相

**活跃度≠健康度。**

MyBatis社区的issue分布：

| 问题类型 | 占比 | 本质 |
|--------|------|------|
| XML配置错误 | 25% | **框架自造** |
| ResultMap映射失败 | 20% | **框架自造** |
| OGNL表达式错误 | 15% | **框架自造** |
| 插件冲突 | 10% | **框架自造** |
| 版本兼容问题 | 10% | **框架自造** |
| **真正的SQL/业务问题** | **20%** | **真问题** |

**80%的社区讨论，是在解决框架制造的问题。**

这就是“受害者联盟”：
- 框架制造问题
- 开发者踩坑
- 社区提供解决方案
- 方案产生新问题
- 继续踩坑

**闭环。自洽。繁荣。**

**但无意义。**

---

### 3.2 SQL-First的“无社区”是健康态

**SQL-First范式的issue应该长什么样？**

| 问题类型 | 归属 | 解决方案 |
|--------|------|--------|
| 这个SQL查询慢 | 数据库/SQL | 加索引、改写SQL |
| 事务不回滚 | Spring事务 | 检查@Transactional |
| 参数传错了 | 业务代码 | 修bug |

**没有一个问题是“框架问题”。**

因为没有框架。只有**工具**。

工具不需要社区。锤子不需要社区。螺丝刀不需要社区。

**你用锤子砸到手，你不会去锤子社区发帖。你知道是自己的问题。**

## 四、误解澄清

### ❌ 误解1：SQL-First = 回到JDBC原始社会

**澄清：** 你见过会自己生成INSERT语句、自动填充主键、自动分页、自动映射结果的“原始社会”？

SQL-First是**站在Spring JDBC/Dapper/sqlx的肩膀上**，把那些**值得自动化的重复劳动**交给工具，把**值得掌控的核心逻辑**交还给开发者。

**该自动的自动，该手动的自由。**

---

### ❌ 误解2：手写SQL容易注入

**澄清：** 注入来自**字符串拼接**，不是来自**手写**。

```java
// ❌ 危险
"SELECT * FROM user WHERE name = '" + name + "'"

// ✅ 安全（SQL-First强制）
"SELECT * FROM user WHERE name = ?"
```

**ORM也是用预编译。SQL-First也是用预编译。没有区别。**

---

### ❌ 误解3：复杂映射必须用resultMap

**澄清：** 80%的复杂映射源于**试图用ORM做数据库不该做的事**。

- 把多行聚合为一对多列表？→ **SQL GROUP_CONCAT**
- 把多表join压成嵌套对象？→ **前端做二次处理**

**剩余20%的真正复杂映射，手写代码比配resultMap更清晰。**

## 五、总结：一图胜千言

```
                      SQL优化主权
                           ▲
                           │
                  ● SQL-First（完全掌控）
                           │
                           │
        MyBatis（半遮半掩）●──┬──● JPA（完全丧失）
                           │
                           │
                           │
                     框架自造错误
```

**第一象限：主权在我，错误真实 → SQL-First**
**第二象限：主权丧失，错误混杂 → JPA**
**第三象限：主权半失，错误自造 → MyBatis**

**你的选择，决定了你每天花多少时间解决“框架问题”。**
